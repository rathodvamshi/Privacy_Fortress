Privacy Fortress

Zero-Knowledge AI Chat — Your Identity Stays Yours

The Real-World Problem

Why does this matter?

  • Healthcare: Patient names in prompts can leak into LLM logs and training.
  • Education: Student PII (name, college, roll numbers) sent to third-party AI.
  • Banking & HR: Sensitive data in chat history = compliance risk.
  • One breach = identity exposed — once PII hits the model, you can't "undo" it.

What Makes Us Different

  • Zero-knowledge AI: LLM never sees "Alice" or "KMIT" — only placeholders like [NAME_1], [COLLEGE_1].
  • Two-locker model: Ephemeral session vault (wiped when you leave) + one encrypted profile per user (with consent).
  • Recreate, don't restore: We store one encrypted profile and recreate context when you return.
  • AES-256-GCM: All persistent PII encrypted before DB; decryption only in RAM at display time.

One Line for the Jury

"We use AES-256 to encrypt user identity, keep it outside the AI, and only resolve it at display time."

System Flow

User→Mask→Ephemeral vault→AI (placeholders only)→Unmask in RAM→User sees real text

  • DB stores only masked messages.
  • One encrypted profile per user (Locker 2) with consent.

Tech Stack & Decisions

  • Backend: FastAPI — async, OpenAPI docs.
  • Detection: spaCy NER + regex + RapidFuzz.
  • Ephemeral vault: Redis + TTL (e.g. 30 min).
  • Persistent vault: MongoDB, AES-256-GCM, one doc per user.
  • LLM: Groq (masked input only; model swappable).
  • Auth: JWT (access + refresh).

Two-Locker Architecture

Locker 1 — Ephemeral

  • Per session, Redis, TTL
  • [NAME_1] → Alice, [COLLEGE_1] → KMIT
  • Wiped when session ends

Locker 2 — Persistent

  • Per user, MongoDB, AES-256
  • One blob: { name, college, email }
  • Until "Forget me"
  • Recreate session from it

Proof: Working Solution

  • Live API: POST /api/chat — masked to AI, unmasked to you.
  • Vault: GET/PUT /api/vault/consent, PUT /api/vault/profile, DELETE /api/vault/profile (Forget me).
  • Security: GET /health/security — encryption & vault status.
